## System Overview

Mission Control is an automated multi-agent build pipeline that:
- Scouts product ideas from various sources (Reddit, HN, etc.)
- Routes approved ideas through a 4-stage agent pipeline
- Builds, integrates, and deploys working applications
- Tracks progress via a real-time dashboard

### Core Components

```
┌─────────────────┐     ┌──────────────────┐     ┌─────────────────┐
│   Idea Scout    │────▶│  Idea Inbox      │────▶│  Build Pipeline │
│   (Automated)   │     │  (Manual Review) │     │  (4 Agents)     │
└─────────────────┘     └──────────────────┘     └────────┬────────┘
                                                          │
                    ┌─────────────────────────────────────┘
                    ▼
           ┌─────────────────┐
           │   Deployment    │
           │  (Vercel/GitHub)│
           └────────┬────────┘
                    │
                    ▼
           ┌─────────────────┐
           │    Dashboard    │
           │  (Monitoring)   │
           └─────────────────┘
```

## Data Schema (Convex)

### Primary Collection: `ideas`

**CRITICAL:** The dashboard and build monitor rely on these exact field names.

```typescript
interface Idea {
  // System Fields (auto-generated by Convex)
  _id: string;                    // Unique identifier (e.g., "k172t5d39ecg2j1zawc40kcmed855q6w")
  _creationTime: number;          // Unix timestamp in milliseconds

  // Core Fields (required)
  title: string;                  // Product name
  description: string;            // Product description
  status: IdeaStatus;             // Current pipeline status (see below)
  potential: 'low' | 'medium' | 'high' | 'moonshot';  // Business potential

  // Categorization
  tags: string[];                 // Array of tags (e.g., ["ai", "productivity"])
  targetAudience: string;         // Who is this for?
  source: string;                 // Where did this idea come from?

  // Scope & Planning
  mvpScope: string;               // What should the MVP include?

  // Timestamps
  createdAt: number;              // When idea was created
  approvedAt?: number;            // When idea was approved (if applicable)
  deployedAt?: number;            // When idea was deployed (if applicable)

  // Build Tracking (optional, set by build pipeline)
  buildStage?: BuildStage;        // Current build stage
  taskStatus?: TaskStatus;        // Kanban column status (legacy, use 'status')
  agentSpawns?: AgentSpawn[];     // Track agent execution
  deployedUrl?: string;           // Live deployment URL
  githubRepoUrl?: string;         // GitHub repository URL
  outputLocation?: string;        // Local build directory path
}
```

### Status Values

**CRITICAL:** These are the ONLY valid values for the `status` field:

| Status | Meaning | Dashboard Column |
|--------|---------|------------------|
| `scouted` | New idea, needs review | Backlog |
| `approved` | Approved, waiting for build | Assigned |
| `building` | Currently being built | In Progress |
| `built` | Code complete, needs deploy | Built |
| `deployed` | Live and deployed | Deployed |
| `done` | Complete | Deployed |
| `failed` | Build/deploy failed | Blocked |
| `rejected` | Idea rejected | Blocked |
| `backlog` | Deferred | Backlog |

**⚠️ WARNING:** The dashboard maps these status values to kanban columns. Changing status values requires updating the dashboard mapping.

### Build Stage Values

Used by build monitor to track agent progress (optional field):

| BuildStage | Icon | Description |
|------------|------|-------------|
| `not_started` | ⚪ | No agents have run yet |
| `agents_spawning` | 🚀 | Agents are being initialized |
| `agents_working` | 🔨 | Agents are actively building |
| `building_locally` | 💻 | Local build in progress |
| `pushing_to_github` | 📦 | Code being pushed to GitHub |
| `deploying_to_vercel` | 🚢 | Deploying to Vercel |
| `completed` | ✅ | All stages complete |
| `failed` | ❌ | Build or deploy failed |

### Agent Spawn Tracking

```typescript
interface AgentSpawn {
  agentId: string;           // e.g., "forge", "pixel", "echo"
  agentName: string;         // Human-readable name
  status: SpawnStatus;       // see below
  spawnedAt: number;         // Timestamp
  completedAt?: number;      // When finished
  sessionKey?: string;       // OpenClaw session identifier
  spawnCommand?: string;     // Command to manually resume
  spawnError?: string;       // Error message if failed
}

type SpawnStatus = 
  | 'spawned'      // Initial state
  | 'working'      // Agent is running
  | 'completed'    // Successfully finished
  | 'failed'       // Error occurred
  | 'manual_intervention_required';  // Needs human help
```

## Build Pipeline Architecture

### Pipeline Flow

```
Approved Idea
     │
     ▼
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│  Forge  │───▶│  Pixel  │───▶│  Echo   │───▶│Integrator│
│(Arch)   │    │(Design) │    │ (Copy)  │    │ (Build)  │
└────┬────┘    └────┬────┘    └────┬────┘    └────┬────┘
     │              │              │              │
     └──────────────┴──────────────┴──────────────┘
                    │
                    ▼
            ┌─────────────┐
            │  Deployment │
            │(Vercel/GitHub)│
            └─────────────┘
```

### Agent Execution

Each agent:
1. Reads inputs from previous agents (if any)
2. Creates output in `{BUILDS_DIR}/{ideaId}/{agentName}/`
3. Writes `COMPLETION.md` when done
4. Updates task file with status

**Directory Structure:**
```
{BUILDS_DIR}/{ideaId}/
├── forge/
│   ├── COMPLETION.md          # Required: Signals completion
│   ├── forge-task.json        # Tracking file
│   └── ...agent outputs...
├── pixel/
│   ├── COMPLETION.md
│   ├── pixel-task.json
│   └── ...
├── echo/
│   ├── COMPLETION.md
│   └── ...
└── integrator/
    └── final/
        ├── COMPLETION.md
        └── ...integrated code...
```

### Build Monitor Logic

**File:** `scripts/build-monitor.js`

**Main Loop (runs every 10 min via cron):**

```javascript
async function main() {
  // 1. Process ideas currently in 'building' status
  const buildingCount = await processBuilding();
  
  // 2. If capacity available, pick up next 'approved' idea
  if (buildingCount < MAX_CONCURRENT_BUILDS) {
    pickNextApproved();
  }
}
```

**processBuilding():**
1. Query ideas with `status: 'building'`
2. For each idea, check agent completion status
3. Spawn next agent if previous completed
4. When all agents done, spawn integrator
5. When integrator done, deploy
6. Mark as `done` when deployed

**pickNextApproved():**
1. Query ideas with `status: 'approved'`
2. Take first one
3. Call `ideas:markBuilding({ ideaId })` to change status

## Agent Roles & Responsibilities

### Forge (Architecture Agent)

**Role:** System design and technical architecture
**Timeout:** 1800s (30 min)
**Model:** anthropic/claude-opus-4-6

**Responsibilities:**
- Choose tech stack
- Design database schema
- Create project structure
- Write initial config files
- Define API contracts

**Outputs:**
- `architecture.md`
- `tech-stack.md`
- Initial code scaffolding

### Pixel (Design Agent)

**Role:** UI/UX design and component specifications
**Timeout:** 2400s (40 min)
**Model:** anthropic/claude-sonnet-4-6

**Responsibilities:**
- Create design system
- Specify colors, typography, spacing
- Design component layouts
- Ensure responsive design
- Apply "Ralph Loop" (simplify aggressively)

**Outputs:**
- `design-system.md`
- `components.md`
- Mockup descriptions

### Echo (Copy Agent)

**Role:** All text content and messaging
**Timeout:** 900s (15 min)
**Model:** moonshot/kimi-k2.5

**Responsibilities:**
- Write product tagline
- Create landing page copy
- Write feature descriptions
- Create onboarding text
- Write UI labels and error messages

**Outputs:**
- `copy.md`
- Content guidelines

### Integrator (Full-Stack Developer)

**Role:** Combine all agent outputs into working application
**Timeout:** 2400s (40 min)
**Model:** anthropic/claude-opus-4-6

**Responsibilities:**
- Read all agent outputs
- Create unified codebase
- Implement all features
- Ensure build passes
- Add Vercel Analytics
- Make deployment-ready

**Outputs:**
- `final/` directory with complete app
- `COMPLETION.md` with deployment instructions

## Status Flow & Transitions

### Normal Flow

```
scouted → approved → building → built → deployed → done
```

### State Transitions

| From | To | Trigger | Actor |
|------|-----|---------|-------|
| `scouted` | `approved` | User clicks "Approve" | Human |
| `scouted` | `rejected` | User clicks "Reject" | Human |
| `approved` | `building` | Build monitor picks up idea | System |
| `building` | `built` | Integrator completes | System |
| `built` | `deployed` | Vercel deploy succeeds | System |
| `deployed` | `done` | Screenshot captured | System |
| Any | `failed` | Error in pipeline | System |

### Recovery Paths

**Failed builds:**
1. Fix issue manually
2. Reset status to `approved` or `building`
3. Build monitor will retry

**Stale agents:**
- Build monitor detects tasks > 2 hours old
- Automatically re-spawns stale agents

## Common Issues & Debugging

### Issue: Dashboard shows "0" for all counts

**Cause:** Schema mismatch between dashboard and Convex data

**Solution:**
- Check that dashboard uses `status` field (not `taskStatus`)
- Verify status values match the mapping table
- See ARCHITECTURE.md "Status Mapping" section

### Issue: Ideas not moving through pipeline

**Cause:** Build monitor not running or can't connect to Convex

**Debug steps:**
```bash
# Check if build monitor is in cron
crontab -l | grep build-monitor

# Run manually to see errors
cd ~/.openclaw/workspace/mission-control
node scripts/build-monitor.js

# Check Convex connection
npx convex run ideas:list | head -5
```

### Issue: Agents not spawning

**Cause:** OpenClaw not configured or agents not defined

**Debug steps:**
```bash
# Check agents exist
openclaw agent list

# Check agent configs
cat ~/.openclaw/workspace/mission-control/agents/forge/agent/AGENTS.md
```

### Issue: Deployment fails

**Cause:** Missing Vercel token or GitHub token

**Debug steps:**
```bash
# Check secrets file
cat ~/.openclaw/workspace/mission-control/config/secrets.json

# Verify Vercel token works
npx vercel --token $(cat config/secrets.json | jq -r .vercel.token) list
```

### Issue: Dashboard shows old data

**Solution:**
- Dashboard caches Convex queries
- Refresh browser
- Check Convex dashboard for actual data

### Issue: Dashboard shows wrong idea count (e.g., 83 instead of 21)

**Root Cause:** Vercel deployment connected to wrong Convex database

**What happened (2026-04-24):**
- Two Convex deployments existed:
  - `beloved-giraffe-115` (dev): 83 ideas
  - `flexible-newt-666` (prod): 21 ideas
- Vercel was using `beloved-giraffe-115` instead of production
- `NEXT_PUBLIC_CONVEX_URL` was baked into build at compile time

**Fix:**
```bash
# 1. Update Vercel env vars to use production Convex
cd dashboard
vercel env rm NEXT_PUBLIC_CONVEX_URL production --yes
vercel env add NEXT_PUBLIC_CONVEX_URL production "https://flexible-newt-666.convex.cloud"
vercel env rm CONVEX_DEPLOYMENT production --yes  
vercel env add CONVEX_DEPLOYMENT production "prod:flexible-newt-666"

# 2. Clear build cache and force rebuild
rm -rf .next
vercel deploy --prod --force

# 3. Reassign alias
vercel alias set <new-deployment-url> mission-control-n8garvie-woad.vercel.app
```

**Prevention:**
- Always verify Convex URL in Vercel dashboard matches production
- Check `npx convex dev` vs `npx convex deploy` contexts
- Document which Convex deployment is "production" in .env files
